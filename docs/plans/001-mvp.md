# Plan 001 — MVP: Chess Coaching Product

Status: **approved, not yet started**
Owner: barathkrishna
Last updated: 2026-04-15

## Product scope

A locally-runnable chess coaching product centered on an interactive chessboard. The user can:

1. Upload a PGN of a game and review it move-by-move on the board, with engine evaluation, weakness classification, and natural-language explanations for each mistake.
2. Play a full game against a human-like opponent (Maia) at a chosen rating.
3. Jump from a just-played game directly into the review flow.
4. See a dashboard of their aggregated weakness profile across all analyzed games.

### Hard constraints

- **No image inputs anywhere.** All inputs are either plain-text PGN (standard chess notation), FEN strings, or direct interaction with the live chessground board. There is no screenshot upload, no OCR, no computer vision, no image analysis at any point in the pipeline. Boards are rendered in the browser by chessground (DOM/SVG) from structured position data.
- **Explanations never contradict the engine.** The classifier detects motifs; Stockfish produces ground-truth PV and eval; the LLM only paraphrases and teaches. Any explanation that contradicts the engine's best line is a bug.
- **Locally runnable end-to-end.** `make serve` boots the whole product. Slice 3 explanations default to a local open-source model through Ollama; hosted API providers are optional later.
- **Reproducible ML.** Every training run has a fixed seed, a checked-in config, and a recorded eval result.

### Out of scope for MVP (explicitly deferred)

- Multi-user auth (single local user, hardcoded ID).
- Puzzle generation from user games.
- Opponent prep / style embeddings / similarity search.
- Cloud deployment, payments, sharing.
- Training a large from-scratch classifier — MVP uses a pragmatic hybrid (heuristics + small learned model, see Slice 6).

## Data flow (upload path)

1. User uploads PGN on the web → `POST /api/games`.
2. API parses PGN with `python-chess`, stores game in SQLite.
3. For each move: Stockfish evaluates position before and after. Compute eval delta.
4. If delta exceeds threshold → classifier assigns motif(s).
5. Motifs + PV + eval delta + FEN → explanation layer → local/API LLM → cached text.
6. Full annotated game returned to frontend, rendered move-by-move on chessground.
7. Weakness counters written to profile store.

## Slices

Each slice ends with a demoable state. `make serve` should work at every slice boundary.

### Slice 0 — Scaffold

**Build:**
- Directory structure per the project layout.
- `CLAUDE.md` with locked decisions and the hard constraints above.
- `pyproject.toml` (uv), `Makefile` with `ingest/train/serve/check/demo` stubs, `.gitignore`.

- `make serve` boots both; localhost:3000 shows the board.
- `make check` runs ruff + mypy + pytest (all green on empty project).

**Done when:** `make serve` → browser shows an interactive chessboard you can move pieces on (no engine behind it yet).

### Slice 1 — Engine wrapper + raw review

**Build:**
- `chess_ml/engine/stockfish.py`: async wrapper around Stockfish binary via UCI. API: `evaluate(fen, depth) → {eval_cp, pv, best_move}`. Worker pool so multiple positions analyze in parallel.
- `chess_ml/ingestion/pgn.py`: parse uploaded PGN to a list of `(move_number, fen_before, move, fen_after)`.
- `chess_ml/api/games.py`: `POST /api/games` accepts PGN, returns annotated game JSON (move list with engine evals).
- Frontend: upload screen (drag-drop PGN text file) → review screen. Review screen has chessground on the left, move list + eval bar + current-move panel on the right. Click a move → board jumps, eval bar updates.

**Done when:** you can upload a real PGN of one of your games and step through it on the board with real Stockfish eval per move. No motifs or explanations yet — just ground truth.

**Test:** pytest unit test for PGN parser and engine wrapper (with a tiny FEN fixture). Manual: upload a known short game, verify eval matches Lichess analysis within a few centipawns.

### Slice 2 — Classifier v0 (hybrid, heuristic-first)

**Build:**
- `chess_ml/classifier/motifs.py`: heuristic motif detectors driven by engine output:
  - **Hanging piece** — piece moves to or remains on a square where it's attacked more than defended, eval delta > 200cp.
  - **Missed tactic** — engine's best move was >300cp better than the played move.
  - **Allowed tactic** — eval drops >300cp on opponent's next move.
  - **Endgame slip** — missed tactic restricted to ≤7 pieces and tablebase-relevant positions.
  - **Opening inaccuracy** — first 15 moves, eval drop 50–150cp.
- These are rules, not learned. That's the point of v0: fast, debuggable, good enough to build the rest of the pipeline against.
- `chess_ml/classifier/classify.py`: takes annotated game from Slice 1, tags each move with 0..N motifs.
- Wire into `/api/games` so the review response includes motifs per move.
- Frontend: show detected motif chip(s) on each move in the move list, colored by severity.

**Done when:** upload a PGN of a game with real blunders, see the blunders correctly flagged with motif labels on the board.

**Test:** fixture tests with hand-curated positions for each motif. Manual: pick 3 real games, verify flagged blunders match expectations.

### Slice 3 — Explanation layer

**Build:**
- `chess_ml/explanation/prompt.py`: prompt template. System prompt establishes the grounding rule (never contradict engine PV, always reference concrete squares/pieces, teach one lesson per explanation, ≤3 sentences). User prompt is `{fen, last_move, user_move, engine_pv, eval_delta, motifs, game_phase}`.
- `chess_ml/explanation/client.py`: provider adapter. Slice 3 defaults to local Ollama with `qwen3:8b`; hosted Anthropic/Codex adapters remain opt-in for later quality/cost testing.
- `chess_ml/explanation/cache.py`: content-addressed cache keyed by hash of `(fen, user_move, motifs)` → stored in SQLite. Prevents re-billing for the same position twice.
- Wire into `/api/games` so response includes per-move explanations.
- Frontend: explanation panel shows the text for the currently selected move. Loading state while it streams.

**Done when:** for every flagged move, you see a short, grounded teaching explanation that references the actual position.

**Test:** unit test that the cache key is deterministic; fixture check that generated explanations reference the engine's best move (not a hallucinated alternative); cache-hit test skips the API call. Manual: review a game, read the explanations, verify they make sense.

### Slice 4 — Profile store + dashboard

**Build:**
- `chess_ml/profile/schema.py`: SQLite tables — `users`, `games`, `positions`, `motif_occurrences`, `explanations`.
- `chess_ml/profile/aggregate.py`: query that returns per-user weakness histogram (motif counts, motif rate per 100 moves, breakdown by game phase and time control).
- API: `GET /api/profile/me` returns the aggregated profile.
- Frontend: `/dashboard` screen showing weakness histogram (bar chart), recent games list, total games reviewed.

**Done when:** upload 3–5 games, visit the dashboard, see a profile that surfaces a real pattern (e.g., "you hang pieces in the endgame more than anywhere else").

**Test:** integration test that inserts synthetic game data and asserts aggregation output. Manual: upload real games, sanity-check the histogram.

### Slice 5 — Play mode vs Maia

**Build:**
- `chess_ml/engine/maia.py`: wrapper around Lc0 binary loaded with Maia weights. API: `predict_move(fen, rating_band) → move`. Rating bands: 1100, 1500, 1900 to start.
- `scripts/fetch_maia_weights.sh`: downloads Maia weights (not checked into git), called by `make setup`.
- API: `POST /api/play/new` starts a game, `POST /api/play/move` submits user move and returns Maia response, `POST /api/play/resign` ends the game and pipes it through the Slice 1–3 analysis pipeline.
- Frontend: `/play` screen with board, clock, rating selector. Post-game: one-click jump to review screen for the game just played.

**Done when:** you can play a full game against Maia at your chosen rating on localhost, then immediately review it with motif detection + explanations.

**Test:** unit test on the Maia wrapper with a fixture position. E2E playwright test: start game, make 5 moves, resign, assert review screen loads. Manual: play a real game.

### Slice 6 — Learned classifier (v1, replaces heuristics where they underperform)

**Build:**
- `chess_ml/ingestion/lichess.py`: download one small slice of Lichess standard rated games (e.g., one day), filter to games with engine evals, extract `(fen, move, eval_before, eval_after, result)` tuples. Label blunders via eval delta thresholds; label motifs via the Slice 2 heuristics (treating heuristics as weak labels).
- `chess_ml/classifier/model.py`: small model — position encoded as 12×8×8 piece planes + metadata (side to move, castling rights, ep, eval-before, move-from, move-to). Architecture: a few Conv blocks → MLP head → multi-label motif logits. ~1–5M params. Deliberately small to train on CPU or a single consumer GPU.
- `chess_ml/classifier/train.py`: training loop with config file, fixed seed, tensorboard/wandb logging, checkpoints.
- Eval suite: held-out Lichess positions, hand-curated hard cases, comparison against the v0 heuristic baseline.
- Wire the trained model into the classifier as an ensemble with the heuristics (heuristics catch obvious cases deterministically; model adds subtler motifs).

**Done when:** the model beats the heuristic baseline on the curated eval set on at least 2 motifs, and the API uses the ensemble.

**Test:** training reproducibility test (seeded run produces same loss to within tolerance), eval-set metrics check.

### Slice 7 — Polish + end-to-end test

**Build:**
- Playwright e2e test covering the full critical path: upload PGN → review → dashboard → play vs Maia → post-game review.
- Loading states, error handling on the frontend (bad PGN, engine timeout, API down).
- README with setup steps.
- `make demo` target: seeds the DB with 3 sample games so anyone cloning the repo can see the product immediately.

**Done when:** `make check && make demo && make serve` works on a fresh clone, and the playwright suite passes.

## Risks & open questions

1. **Maia setup friction on macOS.** Lc0 binary + Maia weights is the standard path but it's fiddly. If it turns into a yak-shave, fallback is the `maia_chess` Python package, or using Stockfish at skill level N as a Maia stand-in (worse, but unblocks the play screen). Decide in Slice 5.
2. **Explanation quality and latency.** Every flagged move can hit the local Ollama model. Measure local latency and output quality after Slice 3; hosted APIs remain a later fallback if local explanations are too weak.
3. **Classifier v0 false positives.** Heuristics will over-flag in complex positions. Mitigation: require eval delta > threshold AND motif match before surfacing. Tune after Slice 2 on real games.
4. **Lichess data volume.** One month of Lichess games is ~30GB compressed. We only need a slice. Pick a small time window (one day, or one rating band) to keep ingestion fast. Decide in Slice 6.
5. **Stockfish depth vs latency.** Deep enough to be trustworthy (~depth 18–20), fast enough that reviewing a 40-move game doesn't take minutes. Parallelize across moves. Budget: <15s for a full 40-move game review.

## Deliverable at the end of Slice 5

A local app where the user can:

1. Play a full game against a human-like bot at their rating.
2. Review any game (uploaded or just-played) move by move on a real chessboard.
3. See each mistake flagged with a motif and read a short coaching explanation grounded in the engine's best line.
4. Visit a dashboard that shows their cumulative weakness profile.

Slices 6 and 7 upgrade the classifier and harden the product, but the core experience is real and playable at the end of Slice 5.
